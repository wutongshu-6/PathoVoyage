### 1.3 `analysis.html` 内核心 JavaScript 概念（单页大脚本）

以下名称与职责为展示时可强调的**核心抽象**（实现均在页面内联 IIFE 中）：

1. `**loadCaseConfig(case_id)`**
  - 用 `fetch`（经 `loadJson`）加载 `case_config_*.json`。  
  - 解析失败或 `custom` 病例时有兜底配置。  
  - 将配置应用到页面标题、资源路径、并加载 `analysis_script` 指向的流程 JSON。
2. `**executeScript(script, clearLog, mode)`** + `**scheduleNext`**（及暂停/继续）
  - 把 `analysis_*.json` 解析为**步骤队列**。  
  - 按 `delay` 用 `setTimeout` 串联执行；支持 `coarse` / `fine` 模式与日志清空策略。  
  - 根据步骤 `type` 分发到：`appendLog`、`show_initial_heatmap`、`highlight_patch`、`show_patch_modal`、进度条与流程子步骤更新等。
3. `**resolveAssetUrl` / `joinPath` / `withNoCache`**
  - 统一静态资源 URL，避免缓存导致热力图错误。
4. **热力图与 Patch UI**
  - 在全切片容器上叠加热力图层、控制透明度；Patch 标记、弹窗内主图与小热力图、图库模态框、滚轮缩放与灯箱等。
5. `**buildReportViewUrl` / `onReportClick` / `downloadAnalysisReport`**
  - 生成报告页 URL（`case_id` 等查询参数）、加载中遮罩延时跳转、或下载本地 `diagnostic_report.pdf`。
6. `**window.PathoAnalysis`**
  - 对外暴露少量接口（如 `appendLog`、`loadCaseConfig`、`executeScript`），便于调试或扩展。

## 二、前端核心逻辑讲解（口头/幻灯可用）

1. **架构形态**
  当前智能分析页是 **单页应用式** 的静态站点：不依赖 Node 构建运行时，由浏览器直接加载 HTML/CSS/JS，通过 `**fetch` 读取 JSON** 与静态图片/PDF。
2. **“AI 分析”的演示实现**
  真实模型推理**不在此仓库的前端代码中执行**。前端通过 `**analysis_*.json` 剧本** 模拟时间线：日志逐条打印、热力图分步揭示、Patch 高亮与弹窗顺序均由 JSON 声明，`**executeScript` 是解释器**。
3. **病例如何切换**
  URL 查询参数（如 `case_id`）决定加载哪一份 `case_config_*.json`，进而决定使用哪套 `analysis_*.json` 与资源路径。
4. **与后端的边界**
  若未来接入真实服务：前端应把 `loadCaseConfig` / 流程进度改为调用 **REST API**；`analysis_*.json` 可改为由服务端根据任务 ID 返回，或仅保留为离线演示模式。

## 三、核心后端代码（仓库现状）

### 3.1 结论

**本仓库（`MajesticAdmin-Free-Bootstrap-Admin-Template/template`）不包含可部署的 Web 后端**（无 Flask/Django/FastAPI/Express 等应用代码）。  
“智能分析”流程由 **静态 JSON + 前端定时器** 驱动；**HTTP 服务器**仅用于提供静态文件（如 `python -m http.server`），**不参与业务计算**。

### 3.2 仓库内与“后端”最接近的内容


| 内容                                              | 说明                                                                                                                |
| ----------------------------------------------- | ----------------------------------------------------------------------------------------------------------------- |
| `**assets/images/generate_patch_manifests.py`** | **离线辅助脚本**：扫描 `all_patches`、`crc_all_patches` 目录下的 JPG，生成 `manifest.json`。因静态站点无法列目录，需开发者本地运行。属于**工具链**，不是线上 API。 |

## 四、补充：真实病理 AI 平台常见后端（供展示“应有后端”）

以下**不是本仓库文件**，可作为答辩时“若系统产品化，后端应包含什么”的**标准答案提纲**。

### 4.1 典型模块划分


| 模块                | 职责                                                                              |
| ----------------- | ------------------------------------------------------------------------------- |
| **API 网关 / 应用服务** | 用户鉴权（JWT/Session）、病例与任务 CRUD、上传切片元数据、任务状态查询。                                    |
| **任务队列**          | Celery + Redis / RQ，异步执行“切片预处理、切 Patch、推理”。                                     |
| **推理服务**          | GPU 上加载病理深度学习模型（如全切片→热力图、Patch 分类），可用 **TorchServe、Triton、或独立 Python gRPC 服务**。 |
| **对象存储**          | MinIO / S3 / 本地 NAS，存放 WSI、派生热力图、报告 PDF。                                        |
| **数据库**           | PostgreSQL：用户、病例、任务、报告状态；可选 Mongo 存日志或非结构化字段。                                   |


### 4.2 核心后端“代码形态”示例（讲解用伪接口）

- **创建分析任务**  
`POST /api/cases/{case_id}/analyze`  
返回 `task_id`，异步跑推理流水线。
- **查询任务与结果**  
`GET /api/tasks/{task_id}`  
返回 `status`、`progress`、`heatmap_url`、`patch_results[]`。
- **流式日志（可选）**  
`WebSocket /api/tasks/{task_id}/stream` 或 SSE，替代前端纯 `setTimeout` 剧本。

### 4.3 与当前前端的对接点


| 前端现状                       | 后端化后                         |
| -------------------------- | ---------------------------- |
| `fetch(case_config.json)`  | `GET /api/cases/{id}/config` |
| `fetch(analysis_npc.json)` | 由任务进度推送或分步 `GET` 返回 `steps`  |
| 静态 `assets/images/`*       | 返回带签名的临时 URL 或 CDN 路径        |
